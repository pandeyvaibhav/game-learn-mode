# Product Backlog

Rolling list of features, improvements, and known gaps for **Game Learn Mode**.

**Status:** `todo` · `in-progress` · `done` · `parked`
**Priority:**
- `P0` — blocks a child from using the product on a phone/tablet (ship first)
- `P1` — needed for the product to be good, not just usable
- `P2` — quality / polish / growth
- `P3` — parked or out-of-scope for now

---

## Priority rationale

Target user is a **primary-school child using this self-serve on a phone or tablet**, with lessons generated by LLM agents. That frames the ranking:

1. **Safety gates come before new content.** Since subject agents produce lessons, content-safety + factual-accuracy reviewers must exist before filling out more years — otherwise unsafe or hallucinated content ships at scale.
2. **Content coverage + mobile safety come before features.** An app that only works for Year 3 or breaks on a phone is not shippable, regardless of how many extras it has.
3. **Core learning loop** (Learn → Try → Play → Check + progress feedback) comes before **retention mechanics** (badges, search, offline).
4. **Accessibility with real controls** is P1, not P2 — kids with motor/switch needs are part of the audience.
5. **Polish items** (breadcrumb tint, gold-standard template) only matter once the above is solid.

## How to use this file
- Add new items to the bottom of the relevant section with a fresh ID (e.g. `M6`, `L5`).
- Keep items sorted within each section by priority (P0 first).
- Update `Status` as work progresses; move `done` items to the bottom of their section or archive them.
- Link to specific files/lines using `[label](path)` so entries stay navigable.
- When adding, ask: *does a child on a phone need this to use the product today?* That distinguishes P0 from everything else.
